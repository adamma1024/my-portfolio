Getting Started with React -- React, Here I Come!

15 May, 2019
|

React has been hyped up by all the front-end gurus, and I never had time to look at it (actually I was just lazy). Today! I’m finally jumping into your arms. Come on, let me feel your charm!

Enough talk, let’s open the official docs and get started!

Getting Started

The demo is a small game — not bad, it’s got something going on. Getting Started Project — What? You want me to pick Option 1, the easy way? No way, Option 2 — build it myself!

class ShoppingList extends React.Component {
  render() {
    return (
      <div className="shopping-list">
        <h1>Shopping List for {this.props.name}</h1>
        <ul>
          <li>Instagram</li>
          <li>WhatsApp</li>
          <li>Oculus</li>
        </ul>
      </div>
    );
  }
}
return React.createElement('div', {className: 'shopping-list'},
  React.createElement('h1', /* ... h1 children ... */),
  React.createElement('ul', /* ... ul children ... */)
);

At first glance, React code has a JSX feel to it. It’s a bit unusual, but still very readable. We define a ShoppingList class (the official docs call it a component class/component type) that extends React’s Component parent class, and it also uses props to pass values. Similar to Vue, createElement is comparable to Vue’s new Vue() — calling render to construct the virtual DOM.

}

Parent-child data passing uses props in both React and Vue, and the syntax is quite similar. Let’s hurry up and get to the interactive part.

class Square extends React.Component {
 render() {

 }

The onClick syntax is consistent with native HTML. Now let’s see how React components store data.

class Square extends React.Component {
  constructor(props) {
    super(props);
    this.state = {
      value: null,
    };
  }

  render() {
    return (
      <button
        className="square"
        onClick={() => this.setState({value: 'X'})}
      >
        {this.state.value}
      </button>
    );
  }
}

Whoa, there’s a constructor! Oh my, how come React is so similar to TypeScript? It even calls super (invoking the parent class constructor — my Java background showing). React components are also pretty straightforward: they use the state property to store component data (reminds me of Vuex state). To modify state, you use the setState method. But something’s wrong with this program — shouldn’t it alternate between X and O? Everyone clicks X — what’s the point? Let’s quickly define a toggle to control whether the next move should be ‘X’ or ‘O’.

Every time setState is called in a component, React automatically updates its child components. (This is a very important statement.)

class Board extends React.Component {
  constructor(props) {
    super(props);
    this.state = {
      squares: Array(9).fill(null),
      xIsNext: true,
    };
  }

  handleClick(i) {
    const squares = this.state.squares.slice();
    if (calculateWinner(squares) || squares[i]) {
      return;
    }
    squares[i] = this.state.xIsNext ? 'X' : 'O';
    this.setState({
      squares: squares,
      xIsNext: !this.state.xIsNext,
    });
  }

  renderSquare(i) {
    return (
      <Square
        value={this.state.squares[i]}
        onClick={() => this.handleClick(i)}
      />
    );
  }

  render() {
    const winner = calculateWinner(this.state.squares);
    let status;
    if (winner) {
      status = 'Winner: ' + winner;
    } else {
      status = 'Next player: ' + (this.state.xIsNext ? 'X' : 'O');
    }

    return (
      <div>
        <div className="status">{status}</div>
        <div className="board-row">
          {this.renderSquare(0)}
          {this.renderSquare(1)}
          {this.renderSquare(2)}
        </div>
        <div className="board-row">
          {this.renderSquare(3)}
          {this.renderSquare(4)}
          {this.renderSquare(5)}
        </div>
        <div className="board-row">
          {this.renderSquare(6)}
          {this.renderSquare(7)}
          {this.renderSquare(8)}
        </div>
      </div>
    );
  }
  function calculateWinner(squares) {
    const lines = [
      [0, 1, 2],
      [3, 4, 5],
      [6, 7, 8],
      [0, 3, 6],
      [1, 4, 7],
      [2, 5, 8],
      [0, 4, 8],
      [2, 4, 6],
    ];
    for (let i = 0; i < lines.length; i++) {
      const [a, b, c] = lines[i];
      if (squares[a] && squares[a] === squares[b] && squares[a] === squares[c]) {
        return squares[a];
      }
    }
    return null;
  }
}

Alright, alright, this is getting boring. Let’s just power through to the end. Take a look, have a peek. Okay, nothing much to see. I find this quite boring. The key takeaway is the keyword that keeps appearing in the official docs: immutable data, immutability.

Let’s See What Immutability Is About

The role of immutability is explained in the Performance Optimization section.

//第一种
class CounterButton extends React.Component {
  constructor(props) {
    super(props);
    this.state = {count: 1};
  }

  shouldComponentUpdate(nextProps, nextState) {
    if (this.props.color !== nextProps.color) {
      return true;
    }
    if (this.state.count !== nextState.count) {
      return true;
    }
    return false;
  }

  render() {
    return (
      <button
        color={this.props.color}
        onClick={() => this.setState(state => ({count: state.count + 1}))}>
        Count: {this.state.count}
      </button>
    );
  }
}
//第二种  简洁版  没区别和第一种
class CounterButton extends React.PureComponent {
  constructor(props) {
    super(props);
    this.state = {count: 1};
  }

  render() {
    return (
      <button
        color={this.props.color}
        onClick={() => this.setState(state => ({count: state.count + 1}))}>
        Count: {this.state.count}
      </button>
    );
  }
}

In most cases, you can use React.PureComponent instead of writing your own shouldComponentUpdate. But it only performs a shallow comparison, so if props or state are mutable in some way, shallow comparison will miss changes and you can’t use it. When data structures are complex, things get tricky.

class ListOfWords extends React.PureComponent {
  render() {
    return <div>{this.props.words.join(',')}</div>;
  }
}

class WordAdder extends React.Component {
  constructor(props) {
    super(props);
    this.state = {
      words: ['marklar']
    };
    this.handleClick = this.handleClick.bind(this);
  }

  handleClick() {
    // 这部分代码很糟，而且还有 bug
    const words = this.state.words;
    words.push('marklar');
    this.setState({words: words});
  }

  render() {
    return (
      <div>
        <button onClick={this.handleClick} />
        <ListOfWords words={this.state.words} />
      </div>
    );
  }
}

The problem is that PureComponent only does a simple comparison between the old and new values of this.props.words. Since the handleClick method in WordAdder mutates the same words array, the old and new this.props.words actually reference the same array. Even though the words in the array have changed, the comparison result is the same. As you can see, even though new words need to be rendered, ListOfWords doesn’t get updated.

Simply put, this is a very common problem when using Vue — the data reactivity detection issue. I’ve encountered this many times during extensive Vue development, especially when an object’s property is an array. Since arrays store a reference, the internal properties change but the monitored reference doesn’t. This leads to a data layer - view layer inconsistency! Vue’s solution is to use Vue.set and Vue.delete. You can check my other article TODO111111111.

The Power of Immutable Data

The exciting moment has arrived — how do we solve this problem that constantly plagues us in Vue?

handleClick() {
  this.setState(state => ({
    words: state.words.concat(['marklar'])
  }));
}
//或
handleClick() {
  this.setState(state => ({
    words: [...state.words, 'marklar'],
  }));
};

The idea is to reassign a new array, so the reference comparison will detect the change.

Admittedly, this approach requires a more experienced programmer to use reliably. Using push is so comfortable — not being able to push and having to create a new object is painful. If I explained this to my colleagues, they probably wouldn’t accept it. Well then, let’s introduce them to a new JS library to make them feel like it’s something fancier!

Using Immutable Data Structures

Immutable.js is another solution. It provides immutable, persistent collections through structural sharing:

  • Immutable: Once created, a collection cannot be modified.
  • Persistent: Modifying a collection creates a new collection. The original collection remains valid.
  • Structural Sharing: New collections reuse as much of the previous collection’s structure as possible, minimizing copy operations to improve performance.
const SomeRecord = Immutable.Record({ foo: null });
const x = new SomeRecord({ foo: 'bar' });
const y = x.set('foo', 'baz');
const z = x.set('foo', 'bar');
x === y; // false
x === z; // true

When defining data structures, use Immutable to define the data as immutable data, then use the API to create a new object each time for assignment. Present it this way to your colleagues and it immediately sounds high-tech, haha (I’m definitely the king of showing off!).
