Vue-cli Pros and Cons

A friend once asked me, “What are the advantages of Vue-cli?” I answered:

“The advantage is that it comes with a ton of built-in configuration.”

Then he asked, “What are the disadvantages?” I replied:

“The disadvantage is that it comes with a ton of built-in configuration.”

Sounds like a classic "live by the sword, die by the sword" situation, doesn’t it? That’s exactly how it feels to me.

Vue-cli integrates a massive amount of webpack configuration, with pre-configured loader, babel, and even eslint. For development, you can just write your code and run:

npm run serve

And you can see your page right away. It seems very convenient, but for beginners, it creates a "knowing how but not why" problem. That’s why the Vue official website has bold text in its Vue-cli introduction saying: “We do not recommend that beginners directly use vue-cli.”

Because Vue-cli has done a massive amount of webpack configuration — over 2000 lines of config code:
(Including:

  • webpack-chain config source code — 784 lines
  • vue-cli/packages/@vue/cli-service/lib/Service.js — 423 lines
  • vue-cli/packages/@vue/cli-service/lib/options.js — 145 lines
  • webpack plugin config — 731 lines)

All of this is what makes development so convenient. But when I tried to modify its configuration, I discovered the CLI’s biggest gripe:

Modifying it is truly painful!!

This article focuses on analyzing Vue-cli’s custom webpack configuration:

vue.config.js

After running Vue-cli’s init command, you’ll notice there’s no webpack.config.js — only a vue.config.js. Basically all your webpack configuration goes into this file and gets merged into Vue-cli’s webpack config via webpack-merge.

Simple Configuration

The simplest way to adjust webpack configuration is to provide an object in the configureWebpack option in vue.config.js:

// vue.config.js
module.exports = {
  configureWebpack: {
    plugins: [
      new MyAwesomeWebpackPlugin()
    ]
  }
}

Here’s where it gets tricky!!! Because some variable names in vue.config.js differ from webpack, you might misconfigure things.

Some webpack options are set based on values in vue.config.js, so you can’t modify them directly. For example, you should modify the outputDir option in vue.config.js rather than output.path; you should modify the publicPath option in vue.config.js rather than output.publicPath. This is because the values in vue.config.js are used in multiple places in the configuration to ensure everything works together properly.

Chaining (Advanced)

Modify configuration using webpack-chain.

When you plan to chain-access a specific loader, vue inspect is very helpful.

Modifying Loader Options

For CSS-related loaders, we recommend using css.loaderOptions rather than directly chaining loaders. This is because each CSS file type has multiple rules, and css.loaderOptions ensures you can affect all rules from one place.

// vue.config.js
module.exports = {
  chainWebpack: config => {
    config.module
      .rule('vue')
      .use('vue-loader')
        .loader('vue-loader')
        .tap(options => {
          // modify its options...
          return options
        })
  }
}

Replacing a Loader in a Rule

If you want to replace an existing base loader, for example using vue-svg-loader for inline SVG files instead of loading the file:

// vue.config.js
module.exports = {
  chainWebpack: config => {
    const svgRule = config.module.rule('svg')

    // Clear all existing loaders.
    // If you don't do this, the next loader will be appended after the existing loaders.
    svgRule.uses.clear()

    // Add the replacement loader
    svgRule
      .use('vue-svg-loader')
        .loader('vue-svg-loader')
  }
}

Modifying Plugin Options

// vue.config.js
module.exports = {
  chainWebpack: config => {
    config
      .plugin('html')
      .tap(args => {
        return [/* new arguments to pass to html-webpack-plugin's constructor */]
      })
  }
}
// You need to be familiar with the webpack-chain API and read some source code
// to make the most of this option, but it's more expressive and safer than
// directly modifying webpack configuration.

// For example, if you want to change index.html's default path from
// /Users/username/proj/public/index.html to /Users/username/proj/app/templates/index.html,
// you can refer to html-webpack-plugin to see the list of options you can pass in.
// We can pass a new template path in the following configuration:

// vue.config.js
module.exports = {
  chainWebpack: config => {
    config
      .plugin('html')
      .tap(args => {
        args[0].template = '/Users/username/proj/app/templates/index.html'
        return args
      })
  }
}

Inspecting the Project’s Webpack Configuration

Here comes the main event!

//package.json里面加入这句命令
"inspect": "vue-cli-service inspect"

//然后命令行里面
npm run inspect > output.js

You’ll find a new output.js file in your directory. Open it — 1155 lines of code. All the configuration is right there. There are also some more convenient commands:

# 只审查第一条规则
vue-cli-service inspect module.rules.0
# 或者指向一个规则或插件的名字
vue-cli-service inspect --rule vue
vue-cli-service inspect --plugin html
# 或者指向一个规则或插件的名字
vue-cli-service inspect --rules
vue-cli-service inspect --plugins

Of course, the recommended approach is to just use the first command npm run inspect to output everything and then ctrl + F to find what you need to change.
Yes, it’s that tedious.

Here’s an example: if I want to modify the CopyWebpackPlugin plugin’s properties, it takes a five-step process:

  1. npm run inspect > output.js
  2. Find CopyWebpackPlugin’s name (modified by config) and existing parameters in output.js
  3. Modify the parameters in vue.config.js
  4. npm run inspect > output.js again to check if the new parameters were applied, or if you’re confident, skip to step five
  5. npm run build

Let’s walk through the process:

npm run inspect > output.js
//找到CopyWebpackPlugin,注意下面的注释，这是这个插件在vue.config.js里面的新名字 config.plugin('copy')
/* config.plugin('copy') */
  new CopyWebpackPlugin(
    [
      {
        from: 'E:\\web_pro\\nr.os-ui\\public', // 拷贝哪里的文件
        to: 'E:\\web_pro\\nr.os-ui\\dist',     // 拷贝去何方
        toType: 'dir',                         // 新建文件夹么
        ignore: [                              // 忽略哪些文件
          '.DS_Store'
        ]
      }
    ]
  )

What if I want to add files to ignore? Check the CopyWebpackPlugin official docs.

// vue.config.js 里面添加
chainWebpack: config => config.plugin('copy').tap((args) => {
  args[0][0].ignore.push('output/**')  //因为里面是个数组，这个数组是args[0]所以这个参数里面的第一个子元素是args[0][0]
  return args
}),
//添加成功
/* config.plugin('copy') */
  new CopyWebpackPlugin(
    [
      {
        from: 'E:\\web_pro\\nr.os-ui\\public', // 拷贝哪里的文件
        to: 'E:\\web_pro\\nr.os-ui\\dist',     // 拷贝去何方
        toType: 'dir',                         // 新建文件夹么
        ignore: [                              // 忽略哪些文件
          '.DS_Store',
          'output/**'
        ]
      }
    ]
  )
npm run inspect > output.js

Done! This is just so much fun (pain).

Build

Library

First, you should learn about webpack library-related concepts.
Then you can use this command to build what you need:

vue-cli-service build --target lib --name myLib [entry]
# File                     Size                     Gzipped

# dist/myLib.umd.min.js    13.28 kb                 8.42 kb
# dist/myLib.umd.js        20.95 kb                 10.22 kb
# dist/myLib.common.js     20.57 kb                 10.09 kb
# dist/myLib.css           0.33 kb                  0.23 kb

Vue vs. JS/TS Entry Files

When using a .vue file as the entry, your library will directly expose the Vue component itself, since the component is always the default export. However, when using a .js or .ts file as the entry, it may contain named exports, so the library will be exposed as a module. This means your library must be accessed via window.yourLib.default in a UMD build, or via const myLib = require(‘mylib’).default in a CommonJS build. If you don’t have any named exports and want to directly expose the default export, you can use the following webpack configuration in vue.config.js:

module.exports = {
  configureWebpack: {
    output: {
      libraryExport: 'mlLibrary' //你得库的名字
    }
  }
}
