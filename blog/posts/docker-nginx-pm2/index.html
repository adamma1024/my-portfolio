Docker + Nginx + WebHook + Node: One-Click Automated Continuous Deployment

31 Dec, 2018
|

Docker’s cross-platform portability + Nginx proxy + Webhook auto-listening for git push = a one-line command to run an automated continuous deployment system

This article walks you through building the above system from scratch, suitable for static websites or front-end/back-end separated websites.

For non-static websites, see: Docker + Nginx + WebHook Automated Continuous Deployment 2.0

This documents the joy of building from scratch, and the pitfalls I encountered along the way. I hope everyone can avoid them. If you have other questions, feel free to reach out — let’s learn and grow together!

This article was built on an Alibaba Cloud ECS server running Ubuntu 16 64-bit. If you don’t have one, they’re pretty cheap right now with discounts: Alibaba Cloud

If you’re already familiar with Docker and don’t need to understand the concepts, you can jump straight to the Hands-On section.

Docker Installation

The installation tutorial works fine, so I won’t go into detail here. Just follow along and you’ll get it working. Installation Tutorial Mirror Configuration Tutorial

# Helloworld测试
docker run hello-world

If you run the command above and see a screen of output with Hello from Docker! at the top, you’re all set.

Docker Concepts

image

A Docker image is a special filesystem that provides not only the programs, libraries, resources, and configuration files needed for container runtime, but also some runtime configuration parameters (such as anonymous volumes, environment variables, users, etc.). An image contains no dynamic data, and its contents don’t change after being built.

# 常用命令

# 获取镜像
docker pull XXX
# 搜索镜像
docker search XXX
# 删除镜像
docker rmi XXX
# 创建镜像，通常使用Dockfile创建, . 是相对路径的意思，需要吧Dockerfile放到与命令执行目录同级
docker build -t XXX .

# Dockerfile 简易 详细的自己查下
# 从那个镜像获取
FROM nginx:latest
# 执行那些命令
RUN echo '<h1>Hello, Docker!</h1>' > /usr/share/nginx/html/index.html

My summary: An image is essentially a packaged application that bundles configuration, runtime instructions, and the environment together.

container

The relationship between an Image and a Container is like the relationship between a class and an instance in object-oriented programming. An image is a static definition, while a container is the runtime entity of an image. Containers can be created, started, stopped, deleted, paused, etc.

# 常用命令
# 所有命令 ID 使用前三位即可
# 显示所有容器
docker ps -a
# 显示所有容器ID
docker ps -aq
# 关闭/启动/重新启动容器
docker stop/kill/start/restart ID
# 删除容器
docker rm ID
# 删除所有容器（启动和没启动的）
docker rm $(docker ps -aq)
# 进入容器 后面的命令行自己随意，推荐 bash
docker exec -it ID /bin/bash
# -d 后台运行 -p映射端口 前面的是docker的端口，后面的是映射到 服务器的端口
docker run -p 8000:80 -d XXX

My summary: Using a container is like new-ing an image — the image starts working.

Docker Registry

After an image is built, it can easily run on the current host machine. But if you need to use this image on another server, you need a centralized service for storing and distributing images — that’s what Docker Registry is.

Similar to an npm registry, but it stores images.

Docker-compose Installation

If you’ve tried the create image -> create container process yourself, you probably feel it’s not that convenient — actually quite tedious. If one is like this, imagine coordinating multiple containers! Well, docker-compose was born to solve this problem.

Compose is a tool for defining and running multi-container Docker applications. With Compose, you use a YAML file to configure your application’s services. Then, with a single command, you create and start all services from the configuration.

# 快来试试威力吧
# docker-compose.yml
version: '3.1' services:
  mongo:
    image: mongo
    restart: always
    ports:
      - 27017:27017
  mongo-express:
    image: mongo-express
    restart: always
    ports:
      - 8000:8081

Now you only need one command to start all your configured containers:

# 启动
docker-compose up

Nginx Image Installation

Here we’re just pulling the Nginx image with Docker.

# 如果你之前在服务器上装过，你一定知道多头疼
# 这就是docker的威力，只需要一行
docker pull nginx

The real complexity of Nginx isn’t in the installation — it’s in the configuration files, which can be really long and verbose, quite unfriendly for beginners! Fortunately, we don’t need much for this article.

WebHook Configuration

Even if you haven’t heard of Webhook, you’ve probably thought about it: how great would it be if every git push to GitHub could automatically pull new code and restart the service? Webhook is the key to making this happen.

Webhook, as the name suggests, is a network hook. It’s easy to think of Vue lifecycle hooks or EventEmitter — both follow the publish-subscribe pattern. Webhook works the same way: GitHub operations from push to creating branches, forking, and even starring all have corresponding hooks. Here’s the webhook configuration page:

Steps:

  1. Select any project
  2. Go to Settings
  3. Select Webhooks
  4. Create a new webhook
  5. Fill in the URL and secret (you’ll need these in the hands-on section — remember them!)
  6. Choose the data format to receive
  7. Update webhook

Hands-On

⚓️

Alright, after all that talk, it’s finally time for hands-on! Let’s go!

First, add the following two files to your project:

Webhook Usage

# autoDeploy.sh
#!/bin/bash
# deploy-dev.sh
echo Deploy Project # 获取最新版代码

# 拉取代码
git pull

# 下面的先注释，等配置完 docker-compose 和 Nginx 再打开
# 强制重新编译容器
# docker-compose down
# docker-compose up -d --force-recreate --build
const http = require('http')
const createHandler = require('github-webhook-handler') //这个插件使用前先安装 npm i github-webhook-handler -D

function run_cmd(cmd, args, callback) {
  var spawn = require('child_process').spawn;
  var child = spawn(cmd, args);
  var resp = "";
  child.stdout.on('data', function (buffer) {
    resp += buffer.toString();
  });
  child.stdout.on('end', function () {
    console.log('Deploy 完成')
    callback(resp)
  });
}

const handler = createHandler({
    path:'/resume-hook', // url 后缀
    secret:'xxxxxxx' // 你的密码
})

// ❗️ 注意要在阿里云安全组里面添加开放端口
// ❗️ 还要在ubuntu防火墙关闭对应端口
http.createServer((req,res) => {
  handler(req,res,err => {
    res.statusCode = 404
    res.end('no such location')
  })
}).listen(7778, () => {
  console.log('Webhook listen at 7778')
})

handler.on('error',err => {
  console.error('Error',err.message)
})

// 拦截push，执行 Deploy 脚本
handler.on('push', function (event) {
  console.log('Received a push event for %s to %s', event.payload.repository.name, event.payload.ref);
  // 分支判断
  if(event.payload.ref === 'refs/heads/master'){
    console.log('deploy master..')
    run_cmd('sh', ['./autoDeploy.sh'], function(text){ console.log(text) });
  }
})

Don’t forget to open the ports! I was stuck on this issue for ages — a painful lesson.

Build Your Project

I used webpack to build to a dist folder, which will be used in the next step — it gets copied into the Nginx container to serve pages. When configuring docker-compose.yml, don’t forget to change the folder name! Also, make sure to push this folder to GitHub — check if .gitignore has it listed, and remove it if so.

Nginx and docker-compose Configuration

Add nginx/conf/docker.conf to your project with the following contents:

Be careful with spacing — Nginx configuration has very strict formatting requirements. I recommend copying and pasting directly.

# 80端口是 Nginx 容器的，后面会做映射
server {
    listen 80;
    location / {
      root /var/www/html;
      index index.html index.htm;
    }
}

Then add a docker-compose.yml file:

# 这里也有坑，version 是和你的 docker-compose版本有关的，如果报错就试试改成 3 或者 3.1
# 打包后存放的文件夹，我的叫dist！！！ 别忘了修改成自己的
version: '2'
services:
  nginx:
    restart: always
    image: nginx
    ports:
      - 80:80
    volumes:
      - ./nginx/conf/:/etc/nginx/conf.d
      - ./dist/:/var/www/html/

OK, we’re done! Now uncomment the lines at the end of autoDeploy.sh and git push.

Last Step: Run webhook.js

  • SSH into the server, git clone xxx
  • cd xxx
  • node webhook.js (if you don’t want to watch logs, use pm2 daemon to run it)
